## 2. SQL-скрипт

### 2.1 SELECT — отримання даних

```sql
-- Виведено ім'я та імейл всіх акаунтів, які мають статус Online
SELECT username, email
FROM Account
WHERE online = 'Online';
```

**Мета:** отримати список активних гравців (лише `username` та `email`), які зараз перебувають в мережі.  
**Очікуваний результат:** рядки з акаунтами зі статусом `'Online'` (за тестовими даними — Stalker та Legolas).  
**Результат:** запит виконано успішно, повернуто 2 рядки.

### 2.3 UPDATE — зміна даних

```sql
-- Оновлено кількість гравців до 67 на сервері де id = 5
UPDATE game_server
SET current_characters = 67
WHERE server_id = 5;
SELECT * FROM game_server;
```
<img width="1263" height="308" alt="image" src="https://github.com/user-attachments/assets/e1d93ccc-4eb7-46fc-a47e-cbce624fe1c9" />

**Мета:** відобразити зміну навантаження — на сервері №5 кількість поточних гравців зросла до 67.  
**Очікуваний результат:** у рядку з `server_id = 5` поле `current_characters` змінено з `5` на `67`.  
**Результат:** запит виконано успішно, зміну підтверджено через `SELECT`.

## 4. Висновки

У ході лабораторної роботи були відпрацьовані основні DML-операції мови SQL у PostgreSQL на базі даних багатокористувацької RPG-гри.

Виконані `SELECT`-запити з фільтрацією через `WHERE` дозволили отримати лише потрібні записи (активні акаунти та досвідчені персонажі) без зайвих даних. За допомогою `INSERT` до таблиць `Region` та `game_server` були успішно додані нові рядки з перевіркою через `SELECT`. Оператори `UPDATE` із обов'язковим `WHERE` дозволили точково змінити лише потрібні записи, не зачіпаючи інші рядки. Операції `DELETE` також виконувались виключно з умовою `WHERE`, що унеможливило випадкове очищення таблиць.

Усі запити виконано без синтаксичних помилок та порушень обмежень схеми, результати кожної операції перевірено через відповідний `SELECT`.
