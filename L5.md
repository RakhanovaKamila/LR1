# Лабораторна робота 5: Нормалізація бази даних

## Вихідна схема

Схема бази даних містить такі таблиці (з лабораторних робіт 1–4):

- `Account`
- `game_server`
- `Region`
- `Game_Structure`
- `Game_Character`
- `ItemTemplate`
- `Inventory_Slot`
- `WeaponStas`
- `Quest_System`
- `Character_Quests`
- `Structure_ownership`

---

## 1. Функціональні залежності початкової схеми

### Account
- `account_id` → `email, account_password, username, registration_date, online, status, in_game_time`
- `email` → `account_id, username, ...` *(унікальне поле)*
- `username` → `account_id, email, ...` *(унікальне поле)*

### game_server
- `server_id` → `server_name1, ip_address, status, current_characters`

### Region
- `region_id` → `server_id, region_name, region_biome, region_min_level`
- `server_id` → *(посилання на game_server)*

### Game_Structure
- `structure_id` → `region_id, structure_name, structure_position, structure_type, is_buyable, structure_cost`

### Game_Character
- `character_id` → `account_id, server_id, region_id, respawn_structure_id, nickname, character_level, character_health, character_stamina, character_class, character_position`

> ⚠️ **Примітка:** Поле `server_id` у `Game_Character` є **транзитивно надлишковим**: сервер вже визначається через `region_id` → `Region.server_id`. Це є порушенням 3NF і виправлено у лаб. роботі 5 командою `ALTER TABLE Game_Character DROP COLUMN server_id`.

### ItemTemplate
- `item_id` → `item_name, item_type, item_description, item_weight, item_max_stack, rarity, item_cost`

### Inventory_Slot
- `slot_id` → `character_id, item_id, item_amount, is_equipped`
- `(character_id, item_id)` → *(функціональна залежність для бізнес-логіки)*

### WeaponStas
- `item_id` → `damage, weapon_bonus`
- `item_id` — зовнішній ключ до `ItemTemplate`

### Quest_System
- `quest_id` → `item_id, quest_name, quest_desciription, quest_xp`

### Character_Quests
- `(character_id, quest_id)` → `quest_status`

### Structure_ownership
- `(character_id, structure_id)` → `structure_role`

## 2. Перевірка нормальних форм початкової схеми

### 1NF (Перша нормальна форма)

**Критерій:** усі атрибути атомарні, немає повторюваних груп.

Всі таблиці **відповідають 1NF**: жодна колонка не містить масивів чи вкладених структур — усі значення є атомарними скалярами.

### 2NF (Друга нормальна форма)

**Критерій:** таблиця в 1NF і жоден неключовий атрибут не залежить лише від *частини* складеного ключа.

| Таблиця | Ключ | Висновок |
|---|---|---|
| `Character_Quests` | `(character_id, quest_id)` | `quest_status` залежить від обох частин ключа ✅ |
| `Structure_ownership` | `(character_id, structure_id)` | `structure_role` залежить від повного ключа ✅ |
| Решта таблиць | простий `SERIAL` PK | часткових залежностей немає ✅ |

**Висновок:** всі таблиці **відповідають 2NF**.

### 3NF (Третя нормальна форма)

**Критерій:** таблиця в 2NF і жоден неключовий атрибут не залежить від іншого неключового атрибута (немає транзитивних залежностей).

#### ❌ Порушення 3NF: таблиця `Game_Character`

```
character_id → server_id
character_id → region_id
region_id    → server_id   ← ТРАНЗИТИВНА ЗАЛЕЖНІСТЬ
```

`server_id` у `Game_Character` транзитивно залежить від `character_id` через `region_id`, оскільки кожен регіон вже прив'язаний до конкретного сервера в таблиці `Region`. Зберігання `server_id` безпосередньо в `Game_Character` створює:

- **Аномалію оновлення:** якщо регіон переноситься на інший сервер, треба оновлювати і `Region`, і всі рядки `Game_Character`.
- **Аномалію вставки:** при додаванні персонажа можна вказати `server_id`, що суперечить серверу регіону.
- **Надлишковість:** інформація про сервер зберігається двічі.

#### ✅ Решта таблиць — відповідають 3NF

У таблиці `WeaponStas` атрибути `damage` та `weapon_bonus` залежать виключно від `item_id` (PK), транзитивних залежностей немає.

У таблиці `Quest_System` атрибут `item_id` є зовнішнім ключем-посиланням на нагороду, але не неключовим атрибутом, що визначав би інші поля — порушення 3NF відсутнє.

## 3. Нормалізація до 3NF: покрокове виправлення

### Таблиця `Game_Character` — усунення транзитивної залежності

#### Оригінальний дизайн

```sql
CREATE TABLE Game_Character (
    character_id         SERIAL PRIMARY KEY,
    account_id           INT REFERENCES Account(account_id),
    server_id            INT REFERENCES game_server(server_id),  -- надлишкове!
    region_id            INT REFERENCES Region(region_id),
    respawn_structure_id INT REFERENCES Game_Structure(structure_id),
    nickname             VARCHAR(50),
    character_level      INT DEFAULT 0 NOT NULL CHECK (character_level >= 0 AND character_level <= 100),
    character_health     INT DEFAULT 10,
    character_stamina    INT DEFAULT 10,
    character_class      class_type,
    character_position   DECIMAL(12, 4)
);
```

**Проблема:** `server_id` є транзитивно залежним: `character_id → region_id → server_id`.  
Сервер персонажа завжди можна отримати через JOIN з таблицею `Region`.

#### Крок 1 → 2NF

Таблиця вже в 2NF (простий первинний ключ, часткових залежностей немає).

#### Крок 2 → 3NF: видалення транзитивної залежності

Видаляємо надлишкове поле `server_id` з `Game_Character`. Сервер визначається через ланцюжок:

```
Game_Character.region_id → Region.server_id → game_server
```

**Команда ALTER TABLE (виконана в лаб. роботі 5):**

```sql
ALTER TABLE Game_Character
DROP COLUMN server_id;
```

#### Переглянутий дизайн таблиці

```sql
CREATE TABLE Game_Character (
    character_id         SERIAL PRIMARY KEY,
    account_id           INT REFERENCES Account(account_id),
    region_id            INT REFERENCES Region(region_id),
    respawn_structure_id INT REFERENCES Game_Structure(structure_id),
    nickname             VARCHAR(50),
    character_level      INT DEFAULT 0 NOT NULL CHECK (character_level >= 0 AND character_level <= 100),
    character_health     INT DEFAULT 10,
    character_stamina    INT DEFAULT 10,
    character_class      class_type,
    character_position   DECIMAL(12, 4)
);
```

**Пояснення:** тепер кожен атрибут залежить **виключно** від `character_id`. Сервер персонажа отримується запитом:

```sql
SELECT gs.server_name1
FROM Game_Character gc
JOIN Region r USING (region_id)
JOIN game_server gs USING (server_id)
WHERE gc.character_id = 1;
```

### Таблиця `Structure_ownership` — додавання первинного ключа

#### Оригінальний дизайн (до лаб. роботи 5)

```sql
CREATE TABLE Structure_ownership (
    character_id INT REFERENCES Game_Character(character_id),
    structure_id INT REFERENCES Game_Structure(structure_id),
    structure_role structure_role_type
    -- PRIMARY KEY відсутній!
);
```

**Проблема:** без первинного ключа таблиця допускає дублікати рядків, що порушує реляційну модель і унеможливлює коректне визначення функціональних залежностей.

**ФЗ:** `(character_id, structure_id)` → `structure_role`

Оскільки `structure_role` залежить від повного складеного ключа — таблиця відповідає 2NF і 3NF **після додавання PK**.

**Команда ALTER TABLE (виконана в лаб. роботі 5):**

```sql
ALTER TABLE Structure_ownership
ADD PRIMARY KEY (character_id, structure_id);
```

#### Переглянутий дизайн таблиці

```sql
CREATE TABLE Structure_ownership (
    character_id INT REFERENCES Game_Character(character_id),
    structure_id INT REFERENCES Game_Structure(structure_id),
    structure_role structure_role_type,
    PRIMARY KEY (character_id, structure_id)
);
```

## 4. Фінальна схема у 3NF — SQL DDL

```sql
-- TYPES
CREATE TYPE online_status       AS ENUM ('Online', 'Offline', 'Sleeping');
CREATE TYPE account_status      AS ENUM ('Banned', 'OK', 'Muted', 'Penalized');
CREATE TYPE server_status       AS ENUM ('On', 'Off');
CREATE TYPE item_category       AS ENUM ('weapon', 'resource', 'trash');
CREATE TYPE rarity_type         AS ENUM ('Common', 'Uncommon', 'Rare', 'Legendary');
CREATE TYPE bonus_type          AS ENUM ('Strength', 'Accuracy');
CREATE TYPE structure_type1     AS ENUM ('ruin', 'circus', 'church', 'mine');
CREATE TYPE class_type          AS ENUM ('Mage', 'Warrior', 'Archer', 'Berserker');
CREATE TYPE quest_status_type   AS ENUM ('Active', 'Completed', 'Failed');
CREATE TYPE structure_role_type AS ENUM ('Owner', 'Co_owner', 'Guest');

-- Account
CREATE TABLE Account (
    account_id        SERIAL PRIMARY KEY,
    email             VARCHAR(255) NOT NULL UNIQUE,
    account_password  VARCHAR(24) CHECK (LENGTH(account_password) >= 8) NOT NULL,
    username          VARCHAR(24) NOT NULL UNIQUE,
    registration_date TIMESTAMP,
    online            online_status NOT NULL,
    status            account_status NOT NULL,
    in_game_time      INTERVAL
);

-- game_server
CREATE TABLE game_server (
    server_id          SERIAL PRIMARY KEY,
    server_name1       VARCHAR(64) NOT NULL,
    ip_address         VARCHAR(45) NOT NULL,
    status             server_status NOT NULL,
    current_characters INT DEFAULT 0 CHECK (current_characters >= 0)
);

-- Region
CREATE TABLE Region (
    region_id        SERIAL PRIMARY KEY,
    server_id        INT NOT NULL REFERENCES game_server(server_id),
    region_name      VARCHAR(100),
    region_biome     VARCHAR(100),
    region_min_level INT NOT NULL
);

-- Game_Structure
CREATE TABLE Game_Structure (
    structure_id       SERIAL PRIMARY KEY,
    region_id          INT REFERENCES Region(region_id),
    structure_name     VARCHAR(50),
    structure_position DECIMAL(12, 4) NOT NULL DEFAULT 0.0000,
    structure_type     structure_type1 NOT NULL,
    is_buyable         BOOLEAN,
    structure_cost     DECIMAL(10, 2) DEFAULT 0
);

-- Game_Character (3NF: server_id видалено)
CREATE TABLE Game_Character (
    character_id         SERIAL PRIMARY KEY,
    account_id           INT REFERENCES Account(account_id),
    region_id            INT REFERENCES Region(region_id),
    respawn_structure_id INT REFERENCES Game_Structure(structure_id),
    nickname             VARCHAR(50),
    character_level      INT DEFAULT 0 NOT NULL CHECK (character_level >= 0 AND character_level <= 100),
    character_health     INT DEFAULT 10,
    character_stamina    INT DEFAULT 10,
    character_class      class_type,
    character_position   DECIMAL(12, 4)
);

-- ItemTemplate
CREATE TABLE ItemTemplate (
    item_id          SERIAL PRIMARY KEY,
    item_name        VARCHAR(50) NOT NULL,
    item_type        item_category NOT NULL,
    item_description TEXT,
    item_weight      DECIMAL(5, 2) DEFAULT 0.0,
    item_max_stack   INT CHECK (item_max_stack > 0 AND item_max_stack <= 64),
    rarity           rarity_type NOT NULL,
    item_cost        DECIMAL(10, 2)
);

-- Inventory_Slot
CREATE TABLE Inventory_Slot (
    slot_id      SERIAL PRIMARY KEY,
    character_id INT REFERENCES Game_Character(character_id),
    item_id      INT REFERENCES ItemTemplate(item_id),
    item_amount  INT DEFAULT 1 CHECK (item_amount > 0),
    is_equipped  BOOLEAN DEFAULT FALSE
);

-- WeaponStas
CREATE TABLE WeaponStas (
    item_id      INT PRIMARY KEY REFERENCES ItemTemplate(item_id),
    damage       DECIMAL(10, 1),
    weapon_bonus bonus_type
);

-- Quest_System
CREATE TABLE Quest_System (
    quest_id           SERIAL PRIMARY KEY,
    item_id            INT NOT NULL REFERENCES ItemTemplate(item_id),
    quest_name         VARCHAR(50),
    quest_desciription TEXT,
    quest_xp           DECIMAL(10, 2)
);

-- Character_Quests
CREATE TABLE Character_Quests (
    character_id INT REFERENCES Game_Character(character_id),
    quest_id     INT REFERENCES Quest_System(quest_id),
    quest_status quest_status_type NOT NULL,
    PRIMARY KEY (character_id, quest_id)
);

-- Structure_ownership (3NF: PRIMARY KEY додано)
CREATE TABLE Structure_ownership (
    character_id   INT REFERENCES Game_Character(character_id),
    structure_id   INT REFERENCES Game_Structure(structure_id),
    structure_role structure_role_type,
    PRIMARY KEY (character_id, structure_id)
);
```

## 5. Підсумок нормалізації

| Таблиця | 1NF | 2NF | 3NF | Зміни |
|---|:---:|:---:|:---:|---|
| `Account` | ✅ | ✅ | ✅ | — |
| `game_server` | ✅ | ✅ | ✅ | — |
| `Region` | ✅ | ✅ | ✅ | — |
| `Game_Structure` | ✅ | ✅ | ✅ | — |
| `Game_Character` | ✅ | ✅ | ❌→✅ | Видалено `server_id` (транзитивна залежність) |
| `ItemTemplate` | ✅ | ✅ | ✅ | — |
| `Inventory_Slot` | ✅ | ✅ | ✅ | — |
| `WeaponStas` | ✅ | ✅ | ✅ | — |
| `Quest_System` | ✅ | ✅ | ✅ | — |
| `Character_Quests` | ✅ | ✅ | ✅ | — |
| `Structure_ownership` | ✅ | ✅ | ✅ | Додано `PRIMARY KEY (character_id, structure_id)` |

### Виконані команди ALTER TABLE

```sql
-- Усунення транзитивної залежності в Game_Character
ALTER TABLE Game_Character
DROP COLUMN server_id;

-- Додавання відсутнього первинного ключа в Structure_ownership
ALTER TABLE Structure_ownership
ADD PRIMARY KEY (character_id, structure_id);
```

Після нормалізації кожна таблиця схеми відповідає **3NF**: кожен неключовий атрибут залежить виключно від первинного ключа, без часткових або транзитивних залежностей.
