## 1. Блок 1: Агрегатні функції

> Запити з використанням `COUNT`, `SUM`, `MIN`, `MAX`, `GROUP BY` для отримання зведеної статистики.

### Запит 1.1 — Кількість гравців певного класу

```sql
SELECT character_class, COUNT(*) AS total_character
FROM Game_Character
GROUP BY character_class;
```

<img width="585" height="175" alt="image" src="https://github.com/user-attachments/assets/68c26df4-51aa-4a24-b186-151f18854e3b" />

**Пояснення:** Групує всіх персонажів за їхнім класом та підраховує кількість гравців у кожному. Дає загальне уявлення про популярність ігрових класів.

---

### Запит 1.2 — Предмет з максимальною кількістю предметів в одному стаку

```sql
SELECT item_name, item_max_stack
FROM ItemTemplate
WHERE item_max_stack = (
    SELECT MAX(item_max_stack) FROM ItemTemplate
);
```

<img width="585" height="178" alt="image" src="https://github.com/user-attachments/assets/90acc80a-2c4c-481d-9f23-a83827df83df" />

**Пояснення:** Знаходить предмет(и), які можна скласти у найбільшу кількість в одному слоті інвентарю. `MAX` обчислює найбільше значення `item_max_stack` серед усіх предметів.

---

### Запит 1.3 — Предмет з найменшою вагою

```sql
SELECT item_name, item_weight
FROM ItemTemplate
WHERE item_weight = (
    SELECT MIN(item_weight) FROM ItemTemplate
);
```

<img width="585" height="173" alt="image" src="https://github.com/user-attachments/assets/c7a65168-c882-4d51-b867-784a7e954303" />

**Пояснення:** Знаходить найлегший предмет у базі даних. `MIN` обчислює мінімальне значення ваги серед усіх предметів.

---

### Запит 1.4 — Загальна ціна всіх речей, які мають цінність

```sql
SELECT SUM(item_cost) AS COST_EVERYTHING
FROM ItemTemplate;
```

<img width="585" height="173" alt="image" src="https://github.com/user-attachments/assets/cf49039b-cb0f-47fc-b4c3-b99c9dfc06aa" />

**Пояснення:** Підраховує загальну вартість усіх предметів у грі. `SUM` підсумовує значення `item_cost` по всій таблиці `ItemTemplate`.

## 2. Блок 2: Типи JOIN

> Запити з використанням `FULL JOIN`, `RIGHT JOIN` та `LEFT JOIN` для об'єднання даних з кількох таблиць.

### Запит 2.1 — Хто де знаходиться, а також їх левел (FULL JOIN)

```sql
SELECT r.region_name, gc.nickname, gc.character_level
FROM Region r
FULL JOIN Game_Character gc USING (region_id)
ORDER BY r.region_name;
```

<img width="586" height="207" alt="image" src="https://github.com/user-attachments/assets/6bac9fc5-36ec-46f6-97fe-43aefeff9f21" />

**Пояснення:** Показує одночасно всіх персонажів їх рівень та регіон,  `FULL JOIN` включає рядки з обох таблиць незалежно від наявності збігу.

---

### Запит 2.2 — Гравці на певному сервері (RIGHT JOIN)

```sql
SELECT server_name1, nickname
FROM Game_Character
RIGHT JOIN Game_Server USING (server_id);
```

<img width="585" height="241" alt="image" src="https://github.com/user-attachments/assets/5d0865e1-6971-4806-95bc-6820a35e9789" />

**Пояснення:** Виводить усі сервери разом із персонажами на них. `RIGHT JOIN` гарантує, що в результат потраплять навіть порожні сервери без жодного гравця.

---

### Запит 2.3 — Всі акаунти з їхніми персонажами (LEFT JOIN)

```sql
SELECT 
    a.account_id,
    a.username,
    a.email,
    a.online,
    c.character_id,
    c.nickname,
    c.character_level,
    c.character_class
FROM Account a
LEFT JOIN Game_Character c
    ON a.account_id = c.account_id;
```

<img width="1267" height="269" alt="image" src="https://github.com/user-attachments/assets/ebb3e39f-d035-47fd-8590-18794ed10fb4" />

**Пояснення:** Виводить усі акаунти, навіть ті, що не мають жодного персонажа (поля з `Game_Character` будуть `NULL`). `LEFT JOIN` гарантує, що жоден акаунт не буде пропущений.

## 3. Блок 3: Підзапити

> Запити з підзапитами у `WHERE`, `SELECT` та `HAVING` для складнішої вибірки даних.

### Запит 3.1 — Персонажі, чий рівень вищий за середній рівень усіх гравців (підзапит у WHERE)

```sql
SELECT nickname, character_level, character_class
FROM Game_Character
WHERE character_level > (
    SELECT AVG(character_level) FROM Game_Character
);
```

<img width="1263" height="269" alt="image" src="https://github.com/user-attachments/assets/3f5f9535-8cda-4bbd-ac0f-a58862959f9b" />

**Пояснення:** Підзапит динамічно обчислює середній рівень по всіх персонажах, а зовнішній запит повертає лише тих, хто перевищує цей поріг. При зміні даних результат оновлюється автоматично.

---

### Запит 3.2 — Список усіх предметів із глобальною середньою ціною (підзапит у SELECT)

```sql
SELECT 
    item_name, 
    item_cost,
    (SELECT ROUND(AVG(item_cost), 2) FROM ItemTemplate) AS avg_global_cost
FROM ItemTemplate;
```

<img width="1263" height="272" alt="image" src="https://github.com/user-attachments/assets/eb914391-159b-4504-8d19-e9750f5c180e" />

**Пояснення:** Скалярний підзапит у `SELECT` обчислюється один раз і додається як окрема колонка `avg_global_cost` до кожного рядка. Дозволяє одразу порівняти ціну кожного предмета із загальним середнім.

---

### Запит 3.3 — Класи, за які грає більше одного персонажа (підзапит у HAVING)

```sql
SELECT character_class, COUNT(character_id) AS total_players
FROM Game_Character
GROUP BY character_class
HAVING COUNT(character_id) > 1;
```

<img width="1264" height="269" alt="image" src="https://github.com/user-attachments/assets/c12e8bc6-6b47-4b55-a401-3a8ecb3cf0c4" />

**Пояснення:** Групує персонажів за класом та через `HAVING` відсіює класи з лише одним представником. `COUNT` у `HAVING` виступає як агрегований підзапит для фільтрації груп.

| **JOIN** | Застосовано `FULL JOIN`, `RIGHT JOIN` та `LEFT JOIN` — 3 запити |
| **Підзапити** | Підзапити у `WHERE`, `SELECT` та `HAVING` — 3 запити |
| **Коректність** | Усі запити виконано в PostgreSQL без помилок; результати відповідають структурі схеми |
| **Документація** | Кожен запит супроводжується поясненням мети та логіки виконання |
